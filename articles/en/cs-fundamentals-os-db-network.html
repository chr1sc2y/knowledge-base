Operating Systems: Processes and Threads

Process vs Thread

A process is the basic unit of resource allocation in an operating system and has its own address space; a thread is the basic unit of CPU scheduling, and threads within the same process share that process's address space and resources. Thread switching is generally cheaper than process switching; processes are more isolated (one crashing usually does not affect others), whereas a thread crash can affect the whole process.

Process States

  1. After a program creates a process, it is admitted to the ready state.
  2. The scheduler dispatches the process to the running state, where the processor executes it.
  3. While running: if interrupted it returns to ready; if waiting for I/O it enters the waiting state and returns to ready when the I/O completes; if execution finishes it enters the terminated state.

Daemon, Orphan, and Zombie Processes

  • Orphan: a process whose parent has terminated but which keeps running; it is adopted by the init process.
  • Zombie: after a process terminates via exit(), its memory and resources are freed, but its entry remains in the process table; the parent uses wait() to read the child's exit status and clear those entries.
  • Daemon: a background process that starts with system boot and ends at shutdown; it typically becomes an orphan after its parent terminates.

Databases and Transactions

SQL vs NoSQL

SQL (relational): table-based storage, a predefined schema, structured query language, and transactions with ACID properties (e.g. MySQL, PostgreSQL); high consistency, but as users grow the database can become a bottleneck, requiring sharding, clustering, and consistent hashing to scale, which is challenging.

NoSQL: dynamic schema for unstructured data. Common types:

TypeDescription and Examples
Key-ValueStores key-value pairs in memory, extremely fast, suited to caching and pub/sub (Redis, Memcached, DynamoDB).
ColumnCan scan a few columns directly for analytics; decentralized and easy to scale; suited to time-series and historical records (Cassandra, HBase).
DocumentStores key-value pairs in JSON/XML documents, flexible and schema-less (MongoDB, CouchDB).
GraphStores data as nodes and the relationships (edges) between them, making associations easy to express.
Search EngineLike a document database but builds an inverted index for searchable terms (Elasticsearch), suited to search and typeahead.

Sharding and Selection

Sharding scales horizontally; a shard key (such as UID) decides which node holds which data. Selection commonly considers whether a cache is needed in front of the database (Redis/Memcached), whether read replicas should offload reads, and the sharding/partitioning strategy.

MySQL: MyISAM vs InnoDB

  • InnoDB supports transactions, MyISAM does not — MyISAM performs better for high-volume queries and inserts, while InnoDB suits frequent modifications with higher safety needs.
  • InnoDB supports foreign keys; MyISAM does not.
  • InnoDB supports row-level locking; MyISAM only table-level locking, so InnoDB performs better under multi-user concurrency.

Transactions and ACID

A transaction is a sequence of operations that either completes as a whole or not at all.

  • Atomicity: the transaction's changes act as a single indivisible operation.
  • Consistency: data is consistent before and after the transaction.
  • Isolation: a transaction's intermediate state is invisible to others.
  • Durability: once committed, changes are persisted.

Isolation Levels

A higher isolation level better guarantees integrity and consistency but has a greater impact on concurrency.

LevelCharacteristic
Read UncommittedChanges are visible even before a transaction commits (dirty read).
Read CommittedOnly committed data from other transactions is readable, resolving dirty reads; a common default in SQL Server, Oracle, MySQL.
Repeatable ReadReads data and applies a lock, resolving non-repeatable reads.
SerializableThe highest level; transactions on the same data run sequentially, resolving phantom reads.

Primary, Unique, and Foreign Keys

  • Primary key: the unique identifier of a record, must have a value, one per table, also a unique key by default, and can be a foreign key in another table.
  • Unique key: may be null; a table can have several.
  • Foreign key: a primary key of another table; may be null; a table can have several.

Read Replicas and CAP

With read-write separation, a primary handles writes while several periodically synced read replicas handle reads — especially useful for read-heavy, high-QPS workloads. Distributed systems follow CAP: since partition tolerance is unavoidable, a trade-off must be made between consistency and availability.

  • C — Consistency: all nodes hold the same data at the same time.
  • A — Availability: every request receives a response.
  • P — Partition Tolerance: the system keeps operating when network partitions occur.

Higher consistency needs more frequent (even synchronous) syncing and may sacrifice availability; higher availability may return stale data.

Redis and Caching

Deployment Modes

ModeCharacteristic
StandaloneA single point of failure.
Master/SlaveA single master handles reads and writes and syncs to multiple read-only slaves; a master failure requires manual replacement or promotion, with brief write unavailability.
SentinelAdds sentinels on top of master/slave; if most sentinels cannot ping the master, it is marked down and a failover automatically elects a new master — no manual action, but sentinel metadata must be maintained and detection has latency.
ClusterData is spread across nodes, each holding part of the data with replicas; easy to scale horizontally (redundancy and fault tolerance).

Gossip Protocol, Persistence, and Single Thread

  • Gossip protocol: a decentralized communication protocol; a Redis cluster uses it to propagate node metadata — a node periodically forwards information to randomly chosen nodes, and it gradually spreads.
  • Persistence: RDB snapshots in-memory data to a binary file on a time-based policy (the default); AOF appends each write command (like a binlog) and replays it on restart. A hybrid of both is common.
  • Why single-threaded: as an in-memory database, reads and writes are fast enough that CPU is not the bottleneck — network I/O is; multithreading would require locks for thread safety, adding CPU overhead.

Common Cache Problems

  • Cache avalanche: many cached items expire at once or the cache service goes down, sending all requests to the database. Mitigation: randomize expiry times or keep keys non-expiring, and use a cache cluster so at least one instance is always available.
  • Cache penetration: requested data exists in neither cache nor database, so every request hits the database directly. Mitigation: cache an empty value briefly for missing keys, or use a Bloom filter to check key existence.
  • Cache breakdown: a hot item expires and high-concurrency requests overwhelm the database. Mitigation: use a mutex so only one thread rebuilds the cache while others wait or return a default; keep hot data non-expiring and refresh it asynchronously in the background.

Eviction Policies

  • FIFO: a queue or doubly linked list.
  • LRU (Least Recently Used): data not accessed recently is unlikely to be accessed soon; a linked list handles insert/remove and a HashMap handles lookup.
  • LFU (Least Frequently Used): a min-heap whose top is the least-frequently accessed item; on overflow, the top is removed.

Data Structures and Consistent Hashing

Skip List

An ordered structure built on linked lists with multiple layers: the bottom is a full sorted linked list, and upper layers are progressively sparser. A search starts at the top and descends layer by layer to approach the target quickly, reducing the average number of steps. Redis sorted sets are implemented with skip lists.

Hashing and Collision Handling

A hash function (e.g. MD5, SHA1) maps a key to an integer that is then taken modulo N. Mapping a larger space into a smaller one makes collisions inevitable; two common solutions:

  • Separate chaining: each mapped value is a bucket; on collision, a node is appended to that bucket's linked list.
  • Open addressing: probe for the nearest empty slot (e.g. linear probing) and insert there.

Consistent Hashing

Servers are distributed evenly on a hash ring; a key is hashed onto the ring and assigned to the next server clockwise. Adding or removing a server only remaps the keys that belonged to it, giving incremental remapping with minimal disruption. If the distribution is uneven, mapping each server to several virtual nodes spreads keys more evenly.

Network Protocols

HTTP Components and Status Codes

An HTTP request consists of a method, URI, version, headers, and body. Status codes: 1xx informational, 2xx success, 3xx redirection (301 permanent, 302 temporary), 4xx client error (400/403/404), 5xx server error (500/502).

Request Methods

GET retrieves information; POST creates a resource; PUT replaces a resource and should be idempotent (a repeated PUT has no side effect); DELETE removes a resource.

HTTP Version Evolution

  • HTTP/1.0: by default a short-lived connection per request, closed after the response; Connection: keep-alive can reuse it.
  • HTTP/1.1: introduced persistent connections, allowing multiple requests on one connection, but sequentially.
  • HTTP/2.0: multiplexing over a single connection (asynchronous concurrent requests/responses), binary compression, server push, and HPACK header compression.

HTTPS and Encryption

  • HTTP and HTTPS are both application-layer protocols; HTTPS adds encryption and verification via SSL/TLS.
  • Symmetric encryption: one key for both encryption and decryption; asymmetric encryption: a public key encrypts and a private key decrypts.

The SSL Handshake

  1. The client requests a secure session, asking the server to identify itself.
  2. The server returns its SSL certificate, including its public key.
  3. The browser verifies the certificate; if trusted, it generates a random symmetric key, encrypts it with the public key, and sends it back.
  4. The server decrypts the symmetric key with its private key; both sides then use it to encrypt the connection.

What Happens When a URL Is Entered in a Browser

  1. The browser resolves the domain to an IP: it checks caches (browser/router) first, and on a miss issues a DNS query (a domain is typically mapped to an IP via a DNS A record).
  2. The resulting IP may be a load balancer, which routes the request to the real backend host.
  3. The browser establishes a TCP connection via the three-way handshake; with HTTPS, a TLS handshake also occurs.
  4. The browser sends HTTP requests; the server responds, and the browser renders the page from the retrieved resources.

TCP Three-Way Handshake and Four-Way Termination

Three-way handshake: (1) the client sends SYN; (2) the server replies SYN-ACK; (3) the client replies ACK, establishing the connection. The third step is needed because, without the client's ACK, the server cannot confirm the client is ready to receive data.

Four-way termination: (1) one side sends FIN to close; (2) the other replies ACK; (3) later, that side also sends FIN; (4) the initiator replies ACK to finish. Four steps are needed because closing is bidirectional and each side must confirm all data has been sent and received.

WebSocket

WebSocket provides full-duplex communication over a single TCP connection; HTTP is half-duplex, requiring the client to poll periodically for new messages. WebSocket supports bidirectional communication.

Distributed Systems

The CAP Trade-off

Partition tolerance (P) is unavoidable in distributed systems, so when a partition occurs a choice must be made between C and A: CP preserves consistency by blocking writes to other nodes; AP keeps accepting reads and writes, where data may be overwritten or stale values returned.

Monolithic vs Microservices

ArchitectureProsCons
MonolithicLower latency, simpler development, code reuse.High complexity, a single fault can bring down the whole, code becomes hard to manage at scale, and parallel development conflicts often.
MicroservicesIndependent scaling of a single service, high maintainability (different tech stacks, communicating via RPC/MQ, friendly to CI/CD), and fault isolation.More underlying infrastructure to maintain, slower development, and extra planning for data modeling and communication.

Containers and Kubernetes

  • Image: a template providing the libraries, resources, and configuration a container needs at runtime.
  • Container: a running instance of an image, building an isolated application environment.
  • Kubernetes: a container-orchestration platform that manages and schedules containers, automating packaging, deployment, and monitoring; it supports fault-tolerant restarts, auto-scaling, service discovery/load balancing, configuration management, and health probes.
  • Pod: the basic scheduling unit in Kubernetes, which can contain multiple containers.
  • Node: the host (physical or virtual) on which pods run, requiring a container runtime, kubelet, and kube-proxy.
