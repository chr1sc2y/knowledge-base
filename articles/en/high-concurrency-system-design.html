Design Methodology

The value of system design lies more in the process than in the final answer: the core skills are organizing one's thinking and focusing on what matters. A common four-step framework follows.

Step 1: Clarify Requirements and Scope

  • Functional requirements: which features are included, who uses the system, and the traffic scale (DAU or QPS). Average QPS ≈ DAU / 86,400, and peak QPS is usually estimated as a multiple of the average.
  • Non-functional requirements: availability (e.g. 99.99%), consistency (strong or eventual, reflecting the CAP trade-off), and security (authentication, rate limiting, abuse prevention).
  • Capacity estimation: estimate transaction volume, storage, and peak load from user counts and per-user behavior. Stateless services are generally easier to scale horizontally.

Step 2: High-Level Design

Sketch the overall architecture: clients, API, CDN, load balancer, service layer, database, cache, and messaging middleware. Outline the data flow first, then discuss details.

Step 3: Deep Dive

For key decisions — database choice (relational / NoSQL / object storage / search engine), partitioning and replication (horizontal sharding / vertical partitioning), and load-balancing strategy (round-robin or least-connections for stateless, consistent hashing for stateful) — list multiple options and justify the trade-offs.

Step 4: Wrap-Up

Review the overall design and confirm it covers the requirements and edge cases.

Load Shedding

For extremely concentrated traffic, common layered load-shedding techniques include:

LayerFunction
CDNPre-caches event pages, static assets, and scripts, absorbing most static requests and reducing origin load.
API GatewayA unified entry point handling authentication, IP/user rate limiting, blocklists, and bot detection (CAPTCHA / human verification).
Stateless horizontal scalingServices perform only light logic behind a load balancer and can scale out nearly without limit.
Message Queue (MQ)Buffers messages to smooth peaks, decouples producers from consumers, and supports asynchronous processing.

Rate Limiting and Circuit Breaking

Rate limiting caps requests per unit time to prevent the system from being overwhelmed; a circuit breaker proactively cuts off requests when a downstream anomaly is detected, preventing cascading failure. Message backlog can be handled along three axes: increasing consumer concurrency, optimizing downstream writes (batch inserts), and proactive rate limiting on the producer side.

Caching and Hotspot Handling

Even with sharding, a single extremely hot key (with QPS in the hundreds of thousands) can overwhelm a single node in the cache cluster (a CPU or network-card bottleneck). A common two-tier defense follows.

Option 1: Key Replication

Replicate the hot key into several copies (e.g. key_1 … key_N) distributed by hashing across different physical nodes; on read, select randomly or round-robin among the replicas. The benefit is that read throughput scales linearly with the number of replicas while the application layer stays stateless; the cost is write amplification (one write updates N copies) with millisecond-level lag.

Option 2: Local Cache

Cache the hottest data directly inside the application process (e.g. Caffeine / Guava Cache) with a short TTL, so requests return at the application layer without a network round trip to the remote cache. The benefit is the lowest latency and full protection of the backend; the cost is that it is stateful, consumes application memory (with OOM risk), and may be briefly inconsistent across instances.

Trade-off: ordinary hotspots suit Key Replication (stateless, simpler operations); extreme, viral hotspots — where the bottleneck is often network bandwidth — suit a Local Cache (eliminating network I/O).

Cache-Layer High Availability

A common three-tier protection: (1) cluster sharding plus replicas with automatic failover, keeping the service available; (2) on cache failure, fail fast and trip a circuit breaker, refusing writes from leaking through to the database (consistency first); (3) after data loss in the cache, replay records already persisted via the message queue and rebuild state from "total minus already-processed", ensuring eventual consistency.

Flash-Sale Inventory Deduction

A flash sale is characterized by a very short window, extremely high concurrency (up to millions of QPS), limited stock, no overselling, and a need for fast feedback. The core principle is that flash-sale traffic must not reach the ordinary checkout system, nor hit the database directly.

Overall Path

User → CDN → API Gateway → Flash-sale service (stateless)
                          ↘ Redis (stock & per-user idempotency)
                          ↘ MQ (async order) → Order service → Database

Inventory Deduction: Atomic Cache Script

During the sale, the cache (e.g. Redis) is the source of truth for stock. A single atomic script performs "check stock > 0 → check whether the user already bought → decrement stock → mark the user" in one shot. Relying on single-threaded, whole-script atomicity prevents overselling and duplicate orders while reducing network round trips.

Evolution of Locking Strategies

ApproachProblem
Database pessimistic lock (row lock)The database becomes the bottleneck and is dragged down under high concurrency.
Optimistic lock (CAS / version)High-contention scenarios trigger a retry storm and CPU spikes.
Distributed lockSerializes parallel requests; a single hot key caps throughput.
Lock-free atomic script (recommended baseline)Executes check-and-decrement atomically inside the cache, with no lock contention.
Stock sharding (ultimate optimization)Split total stock into multiple logical keys (e.g. stock:1001:1 … :N) across nodes, raising write throughput almost linearly.

Issues After Sharding

  • Stock fragmentation: when one bucket sells out while others still have stock, use "request re-routing" (the script returns a signal and the application tries the next bucket) instead of complex stock migration.
  • Data consistency: to prevent a successful cache deduction with a failed message send, transactional messaging (send a half message, then deduct, then commit or roll back based on the result) achieves eventual consistency.

Async Persistence and Reliability

After a successful deduction, a message is sent to the MQ and the order service consumes it asynchronously to persist the order, using a unique request ID for idempotency. Slow consumption is sped up by adding partitions/consumers and batch-writing to the database; failures use retries plus a dead-letter queue (DLQ), with periodic reconciliation for repair.

Supplementary Points

  • Watchdog: a background thread automatically renews a distributed lock that is still in use, avoiding expiry mid-operation.
  • Thundering herd: when a hot cache entry expires, many requests hit the database at once; a mutex can let only one thread rebuild the cache while others wait.
  • Idempotency: a lock only guards concurrency, not duplicate submissions from network retries, so a unique business ID check is also needed.

Low-Latency System Design

In scenarios needing microsecond latency and tens of millions in throughput (such as high-frequency market-data processing), the design goals are real-time response and determinism. A common trade-off is "recency over completeness" — stale data is not just useless but harmful.

DimensionCommon Practice
Ingress protocolUDP multicast rather than TCP, to avoid the jitter of retransmission; gaps are detected by sequence number, and on loss the data is marked "stale", a kill switch is tripped, and state is rebuilt via a snapshot channel rather than waiting for a retransmit.
Physical isolationPort mirroring copies the same data stream to a "trading server" (core computation only) and a "capture server" (a sidecar handling writes/cleaning), achieving fault isolation.
Zero-IO hot pathThe trading path uses kernel bypass (e.g. DPDK); the NIC delivers via DMA into user-space memory; it only receives packets, computes, and sends orders — never writing to disk or logging.
Memory modelHot prices use a pre-allocated array (indexed by price for O(1) access), with cold prices falling back to a red-black tree; an object pool pre-allocates nodes and uses array indices instead of pointers, eliminating runtime allocation and improving cache hit rate.
Capacity stabilityAuto-scaling is avoided (too slow to start, scheduling interferes); instead, static partitioning with CPU pinning/affinity is used, with several times the peak resources reserved.
Testing & releaseDeterministic replay: capture raw packets with high-precision timestamps (PCAP) and replay them with the same timing to verify that the new code's internal state and orders match bit-for-bit.

Additionally, padding to align cache lines avoids false sharing; branch-prediction hints (e.g. unlikely()) keep rarely taken branches from stalling the instruction pipeline.

Strong Consistency and Anti-Overselling

In calendar/inventory systems (such as accommodation booking), the top priority is usually strong consistency — overlapping reservations (overselling) must never occur. Such systems are read-heavy, but popular resources can see flash-sale-level writes at peak.

Storage and Data Model

  • Database choice: although NoSQL scales better, to strictly prevent overlapping reservations the ACID transactions and row-level locking (SELECT ... FOR UPDATE) of a relational database are more robust; NoSQL optimistic locking is prone to retry storms under hot contention.
  • One row per day: one row per listing per day provides the finest-grained row locking — reservations for different dates can be written in parallel without blocking each other, greatly increasing concurrency.
  • Sharding: shard by resource ID to keep each resource's transactions self-contained.

Reservation State Machine (Two-Phase)

  1. Hold: open a transaction, lock the date row with SELECT ... FOR UPDATE, insert the order (status PENDING), set the inventory to HELD with an expiry, then commit and release the connection.
  2. Payment: call the third-party payment provider outside the transaction.
  3. Confirm: on success, a callback updates the status to BOOKED; meanwhile a reconciliation job (cron) scans expired PENDING orders, actively queries payment status, completes paid ones, and releases stock for unpaid ones.

Hotspot Optimization (Write-Behind)

For extremely hot resources (such as tens of thousands competing for the same resource within seconds), a hotspot service is placed in front of the database: stock is warmed into the cache, an atomic script intercepts the vast majority of requests and admits only the winners; successful requests are written asynchronously to a message queue and persisted to the database at a smooth rate (eventual consistency), with total stock serving as the final backstop.
