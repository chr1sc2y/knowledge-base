基本模型：Agent = Model + Harness

在工程语境中，Agent 不是单独的语言模型，而是语言模型与一组外壳机制共同构成的运行系统。模型负责解释输入、生成计划、选择工具和组织输出；harness 负责提供工具接口、状态管理、权限控制、人工审批、日志、评测、沙箱和部署运行时。

Agent = Model + Harness Harness = instructions tool interface state machine memory / context workflow graph permission system human approval tracing / logging eval runner sandbox retry / resume deployment runtime

因此，构建 Agent 的过程也可以理解为逐步打磨 harness 的过程。能力边界需要扩大，失控空间需要缩小，行为结果需要变得可验证。

可靠性通常不是由更长的提示词单独获得，而是由工具边界、状态机、评测集、审批机制和可观测性共同形成。

学习与构建路线

一个渐进式路线通常从单 Agent 和函数调用开始，再加入检索、状态、审批、评测和部署。过早引入多 Agent 往往会增加调试难度，因为工具、状态和评测基础尚未稳定。

阶段核心能力工程产物
模型调用提示词、结构化输出、多轮上下文、流式输出可返回 JSON 的单步任务处理器
工具调用函数 schema、工具错误处理、结果压缩、dry-run只读工具、低风险写入工具、工具调用日志
工作流router、planner-executor、evaluator-optimizer、状态机可暂停和恢复的任务流
安全边界权限、审批、沙箱、数据边界、prompt injection 防御高风险动作确认机制和审计记录
评测与观测trace、eval、回归测试、成本和延迟统计失败样本库、golden traces、监控面板
部署运行队列、持久状态、重试、幂等、取消、灰度API 服务、worker、数据库、对象存储和日志系统

开源实现的参考层次

工业级 Agent 系统通常不需要从零实现所有基础设施。更常见的做法是复用开源框架和观测工具，同时保留业务工作流、权限策略、评测标准和审批体验的自主控制。

类别代表项目适用位置
Agent orchestrationOpenAI Agents SDK、LangGraph、Microsoft Agent Framework、CrewAI工具调用、handoff、工作流图、多 Agent 编排
应用平台Dify快速搭建带 UI、RAG、workflow 和模型管理的 LLM 应用
代码执行型 AgentOpenHands学习长任务、沙箱、代码修改、任务回放和执行环境设计
观测与评测Langfuse、Arize Phoenixtrace、prompt 管理、评测、线上样本分析和调试
工具协议Model Context Protocol统一连接数据库、文件系统、GitHub、搜索、浏览器和内部系统

开源框架适合承担通用运行机制，但业务责任边界通常需要单独建模。例如，是否允许发送邮件、修改生产数据、删除文件或调用付费 API，不应仅由通用框架默认行为决定。

从零构建与基于开源构建

当目标是业务 Agent，常见组合是“基于开源构建，核心业务 harness 自主设计”。工具调用、workflow runtime、trace 和 eval 平台可优先复用；业务状态机、权限模型、人工审批、审计日志和验收标准则需要结合具体场景设计。

更适合复用更适合自主设计
Agent SDK、workflow graph、MCP、RAG 基础组件业务 workflow、任务状态、领域工具 schema
Langfuse/Phoenix 等观测与评测平台任务成功标准、失败分类、线上回归样本
容器、队列、对象存储、托管数据库权限升级、审批体验、回滚策略、审计格式
沙箱、浏览器自动化、代码执行环境的成熟底座允许进入上下文的数据范围和高风险动作限制

最难且最关键的部分

工业级 Agent 的主要难点通常不在于调用模型本身，而在于评测、工具权限、状态恢复和可观测性。

  1. 评测体系：缺少 eval 时，提示词、模型或工具变更可能造成隐性退化。任务集需要覆盖正常、边界、恶意、高风险和长上下文样本。
  2. 工具设计与权限边界：工具接口过宽会增加误用风险，接口过碎会降低效率。高风险工具需要 dry-run、确认、审计和回滚路径。
  3. 状态管理与可恢复性：长任务会中断、超时、失败或等待审批。checkpoint、resume、幂等和取消机制决定系统是否可长期运行。
  4. Tracing 与 observability：生产调试需要知道每次运行的输入、工具调用、工具结果、模型输出、成本、延迟和失败位置。
  5. Context engineering：上下文需要被检索、压缩、排序和丢弃。上下文越长并不等于结果越可靠。
  6. Human-in-the-loop：工业系统常采用逐步放权，而非一次性全自动。人工确认和人工接管是可靠性结构的一部分。
Multi-agent 通常不是第一优先级。多数场景可先由单 Agent、明确工具和 workflow graph 覆盖，再在职责边界稳定后引入多 Agent。

练习 Demo：照片工作流 Agent

一个合适的练习项目是“照片工作流 Agent”：输入一个照片目录，系统只读扫描文件结构，识别 RAW、JPEG、HIF、XMP 等类型，生成整理和修图计划，并把移动、删除、覆盖 XMP 等动作标记为需要确认的高风险操作。

步骤练习重点示例产物
只写工具，不接 Agent工具 schema、路径规范化、只读边界scan_directory、summarize_media_files
接单 Agent函数调用、结构化输出、工具结果压缩目录摘要和行动计划
加入审批高风险动作识别、pending 状态、dry-runapproval request
加入状态机created、scanning、planning、waiting、executing、completed、failedSQLite 任务表和工具调用表
加入 trace可回放、可调试、可归档trace.jsonl、plan.md、summary.json
加入 eval越界路径、删除请求、重复执行、工具失败10 到 30 条回归样本

这个 demo 覆盖工具、权限、状态、trace、eval 和恢复路径，比泛用聊天机器人更接近工业级 Agent 的真实问题。

用 Agent 构建 Agent

Agent 可以参与实现 Agent 系统，但需要处在受控工程流程内。适合自动化的部分包括工具 wrapper、schema、eval runner、trace 集成、UI 草图、文档、测试和根据失败 trace 生成回归样本。

需要由人类或明确治理规则决定的部分包括：权限模型、审批边界、任务成功定义、评测标准、数据隔离策略、生产重试上限和高风险动作的最终责任。

需求与验收标准 -> 设计草案 -> 边界确认 -> 最小实现 -> 测试与 eval -> trace 分析 -> 失败样本回归 -> 受限上线

部署路线

早期 demo 可从本地运行和容器化开始，再迁移到云端托管容器。云端部署需要关注 secret 管理、对象存储、日志、任务状态和长任务 worker。

阶段架构形态目标
本地FastAPI、SQLite、本地 runs 目录、本地 eval验证 Agent 行为和 harness 基础
轻量云端 demoAWS App Runner、S3、CloudWatch、Secrets Manager、Postgres获得可访问的 Web API 和持久化产物
准生产ECS Fargate API、ECS Fargate worker、SQS、RDS Postgres、S3、OpenTelemetry、Langfuse/Phoenix支持长任务、重试、幂等、观测和审计
云原生编排Step Functions、Bedrock Agents 或自建 workflow runtime处理更复杂的可审计长流程和 AWS 原生集成

Lambda 适合承载较小的工具动作，但完整 Agent runtime 往往需要更长运行时间、多步状态、审批等待和较完整的 trace，因此容器形态通常更适合早期系统。

来源与说明

本文是对 Agent 工程常见模式、开源框架文档和公开工程文章的结构化整理，仅作技术学习参考。具体系统设计需要结合数据敏感性、权限要求、业务风险和部署环境进行独立评估。

  • OpenAI Agents SDK: Agents、Tools、Handoffs、Guardrails、Tracing 等文档。
  • LangGraph：长任务、状态化 Agent 和 workflow graph 的开源实现。
  • Anthropic: Building Effective Agents 以及关于 context engineering、tool design、eval 和 long-running harness 的工程文章。
  • Langfuse 与 Arize Phoenix：开源 LLM observability 与 evaluation 工具。
  • Model Context Protocol：连接 Agent 与外部工具和数据源的开放协议。
  • AWS App Runner、AWS Fargate 与 AWS Step Functions 文档。
工业级 Agent 的长期积累对象不是单一提示词，而是工具、状态、评测、权限、观测和部署运行时共同构成的 harness。
