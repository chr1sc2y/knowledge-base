设计方法论

系统设计的价值更多体现在设计过程而非最终方案，核心在于组织思路并聚焦关键问题。一种常见的四步框架如下。

第一步：澄清需求与范围

  • 功能性需求：包含哪些功能、谁来使用、流量量级（DAU 或 QPS）。平均 QPS ≈ DAU / 86400，峰值 QPS 通常按平均值的若干倍估算。
  • 非功能性需求：可用性（如 99.99%）、一致性（强一致或最终一致，对应 CAP 取舍）、安全性（鉴权、限流、防刷）。
  • 容量估算：根据用户量与每用户行为估算事务量、存储量与峰值压力。无状态服务通常便于水平扩展。

第二步：高层设计

给出整体架构概览：客户端、API、CDN、负载均衡、服务层、数据库、缓存、消息中间件。先勾勒数据流，再讨论细节。

第三步：深入细节

就数据库选型（关系型 / NoSQL / 对象存储 / 搜索引擎）、分区与复制（水平分片 / 垂直分区）、负载均衡策略（无状态用轮询/最少连接，有状态用一致性哈希）等关键决策列出多方案并说明取舍理由。

第四步：收尾

回顾整体方案，确认是否覆盖了需求与边界情况。

流量削峰体系

面对极端集中的流量，常见的分层削峰手段如下：

层作用
CDN提前缓存活动页、静态资源与脚本，承接大部分静态请求，降低源站压力。
API 网关统一入口，承担鉴权、IP/用户限流、黑名单与人机校验（防机器人）。
无状态服务水平扩容服务只做轻逻辑、负载均衡分发，可近乎无限横向扩展。
消息队列（MQ）通过缓冲消息实现"削峰填谷"，解耦生产者与消费者，支撑异步处理。

限流与熔断

限流（Rate Limiting）限制单位时间的请求数量，防止系统被压垮；熔断（Circuit Breaker）在发现下游异常时主动切断请求，防止连锁崩溃。对消息堆积，可从"提升消费并发 / 优化下游写入（批量写入）/ 生产侧主动限流"三方面处理。

缓存与热点治理

即使做了分片，单一极热 Key（QPS 可达 10 万以上）仍可能打爆缓存集群中的单一节点（CPU 或网卡瓶颈）。常见的两级防御体系如下。

方案一：热点 Key 复制（Key Replication）

将热点 Key 复制为多份（如 key_1 … key_N），由哈希分布到集群的不同物理节点；读取时在副本间随机或轮询选择。优点是读性能随副本数线性扩展、应用层保持无状态；缺点是一次写入需更新多份，存在写放大与毫秒级延迟。

方案二：本地缓存（Local Cache）

在应用进程内（如 Caffeine / Guava Cache）直接缓存极热数据并设置较短 TTL，请求在应用层即可返回，不经过网络访问远端缓存。优点是延迟最低、彻底保护后端；缺点是有状态、占用应用内存（存在 OOM 风险），且多实例间可能短暂不一致。

取舍：常规热点更适合 Key Replication（保持无状态、运维简单）；现象级热点（瓶颈往往在网络带宽）更适合 Local Cache（消除网络 I/O）。

缓存层高可用

一种常见的三层保护：①集群分片 + 多副本 + 自动故障转移，保证服务不中断；②缓存故障时快速失败（fail-fast）并熔断，拒绝写请求穿透到数据库（一致性优先）；③缓存发生数据丢失后，依赖消息队列中已成功落库的记录回放，按"总量 − 已处理量"重建状态，保证最终一致性。

高并发秒杀与库存扣减

秒杀场景的特征是：时间极短、并发极高（可达百万级 QPS）、库存有限、不允许超卖、需快速反馈。核心原则是秒杀流量不能打到普通下单系统，也不能直接打到数据库。

整体链路

用户 → CDN → API 网关 → 秒杀服务（无状态）
                         ↘ Redis（库存 & 用户幂等标记）
                         ↘ MQ（异步下单） → 订单服务 → 数据库

库存扣减：缓存原子脚本

秒杀期间以 Redis 作为库存的 Source of Truth。用一段原子脚本一次性完成"检查库存 > 0 → 检查用户是否已抢购 → 扣减库存 → 标记用户"，借助单线程整体执行的原子性，保证不超卖、不重复下单，同时减少网络往返。

锁策略的演进

方案问题
数据库悲观锁（行锁）数据库成为瓶颈，高并发下会被拖垮。
乐观锁（CAS / 版本号）高冲突场景引发重试风暴，CPU 飙升。
分布式锁把并行请求串行化，单热点 Key 限制吞吐上限。
无锁原子脚本（推荐基准）在缓存内部原子执行查减，无锁竞争开销。
库存分桶（终极优化）将总库存拆为多个逻辑 Key（如 stock:1001:1 … :N）分散到不同节点，写性能近线性提升。

分桶后的衍生问题

  • 库存碎片化：某桶售罄而其他桶仍有货时，采用"请求二次路由"（脚本返回信号，应用层尝试下一个桶），而非复杂的库存搬运。
  • 数据一致性：防止缓存扣减成功但消息发送失败，可用事务型消息（先发 half message，再执行扣减，按结果提交或回滚）实现最终一致性。

异步落库与可靠性

扣减成功后投递消息到 MQ，订单服务异步消费落库，并以唯一请求 ID 做幂等。消费慢时通过增加分区/消费者、批量写入数据库来提速；失败时配合重试与死信队列（DLQ），并通过定时对账修复。

补充知识点

  • 看门狗机制：后台线程为仍在执行的分布式锁自动续期，避免业务超过 TTL。
  • 惊群效应：热点缓存失效瞬间大量请求打到数据库，可用互斥锁只让一个线程重建缓存，其余等待。
  • 幂等性：锁只能挡并发，挡不住网络重试导致的重复提交，需配合唯一业务 ID 校验。

低延迟系统设计

在需要微秒级延迟、千万级吞吐的场景（如高频行情处理）中，设计目标是实时性与确定性。常见取舍是"实时性 > 完整性"——过时的数据不仅无用，反而有害。

维度常见做法
接入协议采用 UDP 组播而非 TCP，避免重传带来的延迟抖动；基于序列号检测丢包，丢包时不等待重传，而是标记数据为"不可信"、触发熔断并通过快照通道重建。
物理隔离用端口镜像将同一份数据流复制给"交易机"（只做核心计算）与"记录机"（旁路负责写盘/清洗），实现故障隔离。
零 IO 主路交易主路使用内核旁路（如 DPDK），网卡经 DMA 直达用户态内存；只收包、算策略、发单，严禁写盘与打日志。
内存模型热区价格用预分配数组（以价格为下标，O(1) 访问），冷区降级红黑树；用对象池预分配节点、以数组下标代替指针，消除运行时内存分配并提升缓存命中率。
容量稳定性拒绝自动扩缩容（启动太慢、调度干扰），改用静态分区与 CPU 核绑定（Pinning/Affinity），并预留数倍峰值资源。
测试发布确定性重放：用带高精度时间戳的原始抓包（PCAP）以相同时序回放，验证新代码产生的内部状态与指令达到比特级一致。

附：通过填充对齐缓存行可避免伪共享（False Sharing）；通过分支预测提示（如 unlikely()）让极少触发的分支不打断指令流水线。

强一致与防超卖

在日历/库存类系统（如住宿预订）中，最高优先级通常是强一致性——绝对不允许重叠预订（超卖）。读多写少，但热门资源在高峰可能出现秒杀级写入。

存储与数据模型

  • 数据库选型：尽管 NoSQL 扩展性更好，但为严格防止重叠预订，关系型数据库的 ACID 事务与行级锁（SELECT ... FOR UPDATE）更稳健；NoSQL 的乐观锁在高并发热点下易引发重试风暴。
  • 按天一行：每个房源每天一行数据，提供最细粒度的行锁——不同日期的预订可并行写入、互不阻塞，显著提升并发度。
  • 分片：按资源 ID 分库分表，保证单资源事务闭环。

预订状态机（两阶段）

  1. 预占：开启事务，SELECT ... FOR UPDATE 锁定日期行，插入订单（状态 PENDING）、将库存置为 HELD 并设置过期时间，随即提交事务释放连接。
  2. 支付：在事务之外调用第三方支付。
  3. 确认：支付成功经回调将状态更新为 BOOKED；同时用对账任务（Cron）扫描过期的 PENDING 订单，主动查询支付状态，已付则补齐、未付则释放库存。

热点优化（Write-Behind）

对极热资源（如短时间内上万人抢同一资源），在数据库前引入热点服务：预热库存到缓存，用原子脚本拦截绝大多数请求，仅放行成功者；成功请求异步写入消息队列，由后端以平滑速率落库（最终一致性），并以总库存作为最终兜底。
