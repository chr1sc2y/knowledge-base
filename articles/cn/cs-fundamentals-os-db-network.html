操作系统：进程与线程

进程与线程

进程是操作系统进行资源分配的基本单位，拥有独立的地址空间；线程是 CPU 调度的基本单位，同一进程内的多个线程共享该进程的地址空间与资源。线程切换的开销通常小于进程；进程之间隔离性更强（一个进程崩溃通常不影响其他进程），而同进程内一个线程崩溃可能影响整个进程。

进程状态

  1. 进程被程序创建后进入就绪（ready）状态。
  2. 调度器将进程派发到运行（running）状态，由处理器执行。
  3. 运行中：若被中断则回到就绪；若等待 I/O 事件则进入等待（waiting），事件完成后回到就绪；若执行完毕则进入终止（terminated）。

守护、孤儿与僵尸进程

  • 孤儿进程：父进程已终止但自身仍在运行的进程，会被 init 进程收养。
  • 僵尸进程：进程通过 exit() 终止后，内存与资源被释放，但其在进程表中的条目仍保留；父进程通过 wait() 读取子进程退出状态并清除这些条目。
  • 守护进程：随系统启动、随系统关闭的后台进程；其父进程终止后通常成为孤儿进程。

数据库与事务

SQL 与 NoSQL

SQL（关系型）：基于表存储、预定义 schema、使用结构化查询语言、具备事务与 ACID 特性（如 MySQL、PostgreSQL），一致性高；用户量增大时数据库可能成为瓶颈，需要分片、集群、一致性哈希来扩展，扩展较有挑战。

NoSQL：动态 schema、面向非结构化数据，常见类型包括：

类型说明与代表
键值（Key-Value）内存中以键值对存储，极快，适合缓存、发布订阅（Redis、Memcached、DynamoDB）。
列式（Column）可只扫描少量列做分析，去中心化、易扩展，适合时序与历史记录（Cassandra、HBase）。
文档（Document）以 JSON/XML 文档存储键值对，灵活无固定 schema（MongoDB、CouchDB）。
图（Graph）以节点与节点间关系（边）存储，便于表达关联关系。
搜索引擎类似文档数据库，但为可搜索词项建立倒排索引（Elasticsearch），适合搜索与联想补全。

分片与选型

分片（Sharding）用于水平扩展，通过分片键（如 UID）决定数据落在哪个节点。选型时常见考量：是否需要在数据库前加缓存（Redis/Memcached）、是否需要读副本分担读请求、以及分片与分区策略。

MySQL：MyISAM 与 InnoDB

  • InnoDB 支持事务，MyISAM 不支持——MyISAM 性能更高、适合高频查询与插入；InnoDB 更适合频繁修改、对安全性要求更高的场景。
  • InnoDB 支持外键，MyISAM 不支持。
  • InnoDB 支持行级锁，MyISAM 只支持表级锁，因此 InnoDB 在多用户并发下性能更好。

事务与 ACID

事务是一组操作，要么整体完成、要么完全不发生。

  • 原子性（Atomicity）：事务的变更如同单一不可分的操作。
  • 一致性（Consistency）：事务前后数据保持一致。
  • 隔离性（Isolation）：事务的中间状态对其他事务不可见。
  • 持久性（Durability）：事务完成后变更被持久化。

隔离级别

隔离级别越高，对数据完整性与一致性的保证越强，但对并发性能影响越大。

级别特征
读未提交（Read Uncommitted）即使事务未提交也能看到其变更（脏读）。
读已提交（Read Committed）只能读到其他事务已提交的数据，解决脏读；SQL Server、Oracle、MySQL 的常见默认。
可重复读（Repeatable Read）读取数据并加锁，解决不可重复读。
串行化（Serializable）最高级别，操作同一数据的事务串行执行，解决幻读。

主键、唯一键、外键

  • 主键：记录的唯一标识，必须有值，一张表只有一个，默认也是唯一键，可作为另一张表的外键。
  • 唯一键：可为空，一张表可有多个。
  • 外键：是另一张表的主键，可为空，一张表可有多个。

读副本与 CAP

读写分离：主库处理写请求，多个定期同步的读副本处理读请求，对读多写少的高 QPS 场景尤其有效。分布式系统需遵循 CAP——由于分区容错不可避免，必须在一致性与可用性之间权衡。

  • C 一致性：所有节点在同一时刻拥有相同数据。
  • A 可用性：每个请求都能得到响应。
  • P 分区容错：网络分区发生时系统仍能运行。

追求高一致性需更频繁地（甚至同步地）同步数据，可能牺牲可用性；追求高可用则可能返回过时数据。

Redis 与缓存

部署模式

模式特征
单机（Standalone）存在单点故障。
主从（Master/Slave）单主处理读写并同步给多个只读从库；主库宕机需手动替换或提升从库，期间可能短暂不可写。
哨兵（Sentinel）在主从基础上增加哨兵；多数哨兵 ping 不通主库则判定下线并自动故障转移、选出新主库，无需人工，但需维护哨兵元数据、判定有延迟。
集群（Cluster）数据分散到多节点，每节点只含部分数据并有副本，易于水平扩展（冗余与容错）。

Gossip 协议、持久化与单线程

  • Gossip 协议：去中心化通信协议，Redis 集群用它传播节点元数据——节点周期性地随机选取若干节点转发信息，信息逐步扩散。
  • 持久化：RDB 按时间策略将内存数据快照为二进制文件（默认）；AOF 追加每条写命令（类似 binlog），重启时重放重建数据。通常二者混合使用。
  • 为何单线程：作为内存数据库，读写足够快，CPU 不是瓶颈、网络 I/O 才是；多线程需要加锁保证线程安全，反而增加 CPU 开销。

缓存常见问题

  • 缓存雪崩：大量缓存同时过期或缓存服务宕机，请求全部打到数据库。对策：为 key 设置随机过期时间或不过期，并用缓存集群保证至少一个实例可用。
  • 缓存穿透：请求的数据在缓存和数据库都不存在，每次都直达数据库。对策：对不存在的 key 缓存空值（短时），或用布隆过滤器判断 key 是否存在。
  • 缓存击穿：某个热点数据过期瞬间，高并发请求压垮数据库。对策：用互斥锁让单个线程重建缓存，其余等待或返回默认值；热点数据不设过期、后台异步更新。

缓存淘汰策略

  • FIFO：队列或双向链表。
  • LRU（最近最少使用）：长期未被访问的数据更可能将来也不被访问；用链表插入/移除节点、HashMap 查询。
  • LFU（最不经常使用）：用最小堆，堆顶为访问频率最低者；容量超限时移除堆顶。

数据结构与一致性哈希

跳表（Skip List）

用链表实现的有序结构，分多层：底层是完整的有序链表，越往上节点越稀疏。查找从顶层开始逐层下沉，从而快速逼近目标、减少平均步数。Redis 的有序集合即基于跳表实现。

哈希与冲突处理

用哈希函数（如 MD5、SHA1）将 key 映射为整数再对 N 取模。把较大值空间映射到较小值空间时冲突不可避免，常见两种解法：

  • 链地址法（Separate Chaining）：每个映射值是一个桶，冲突时在桶对应的链表尾部追加节点。
  • 开放寻址法（Open Addressing）：通过线性探测等方式寻找最近的空槽插入。

一致性哈希

将服务器均匀分布在一个哈希环上，key 哈希到环上后顺时针分配给下一台服务器。增删服务器时，只有原本属于该服务器的部分 key 会被重新映射，实现增量重映射与最小扰动。若分布不均，可为一台服务器映射多个虚拟节点，使 key 分布更均匀。

网络协议

HTTP 组成与状态码

HTTP 请求由方法、URI、版本、首部、正文组成。状态码：1xx 信息、2xx 成功、3xx 重定向（301 永久、302 临时）、4xx 客户端错误（400/403/404）、5xx 服务端错误（500/502）。

请求方法

GET 获取信息；POST 创建资源；PUT 替换资源且应当幂等（重复 PUT 无副作用）；DELETE 删除资源。

HTTP 版本演进

  • HTTP/1.0：默认每个请求一条短连接，响应后关闭；可用 Connection: keep-alive 复用。
  • HTTP/1.1：引入持久连接，可在单连接上发多个请求，但顺序串行。
  • HTTP/2.0：单连接上多路复用（异步并发请求响应）、二进制压缩、服务端推送、HPACK 首部压缩。

HTTPS 与加密

  • HTTP 与 HTTPS 都是应用层协议；HTTPS 通过 SSL/TLS 实现加密与验证。
  • 对称加密：加解密使用同一把密钥；非对称加密：公钥加密、私钥解密。

SSL 握手过程

  1. 客户端请求建立安全会话，要求服务器表明身份。
  2. 服务器返回其 SSL 证书（含公钥）。
  3. 浏览器校验证书；信任则生成随机对称密钥，用公钥加密后发回。
  4. 服务器用私钥解出对称密钥；此后双方用该对称密钥加密通信。

在浏览器中输入一个 URL 后发生了什么

  1. 浏览器先查找域名对应的 IP：先查缓存（浏览器/路由器），未命中则发起 DNS 查询（域名通常通过 DNS A 记录映射到 IP）。
  2. 得到的 IP 可能是负载均衡器，请求会被路由到真正的后端主机。
  3. 浏览器与服务器经三次握手建立 TCP 连接；若用 HTTPS 还会进行 TLS 握手。
  4. 浏览器发送 HTTP 请求，服务器处理并返回响应，浏览器用获取的资源渲染页面。

TCP 三次握手与四次挥手

三次握手：①客户端发送 SYN；②服务器回 SYN-ACK；③客户端回 ACK，连接建立。需要第三次，是因为没有客户端的 ACK，服务器无法确认客户端已准备好接收数据。

四次挥手：①一方发 FIN 请求关闭；②另一方回 ACK；③稍后另一方也发 FIN；④发起方回 ACK 完成关闭。四次是因为关闭是双向的，需各自确认数据收发完毕。

WebSocket

WebSocket 在单条 TCP 连接上提供全双工通信；HTTP 是半双工，只能由客户端轮询查询是否有新消息。WebSocket 支持双向通信。

分布式系统

CAP 取舍

分区容错（P）在分布式系统中不可避免，分区发生时必须在 C 与 A 之间取舍：CP 通过阻塞向其他节点的写来保持一致性；AP 持续接受读写，数据可能被覆盖或返回过时值。

单体与微服务

架构优点缺点
单体（Monolithic）延迟更低、开发更简单、代码可复用。复杂度高、单点故障可拖垮整体、代码量增大后难管理、并行开发易冲突。
微服务（Microservices）可独立扩展单个服务、可维护性强（不同技术栈，通过 RPC/MQ 通信，利于 CI/CD）、故障隔离。需维护更多底层基础设施、开发更慢、需额外规划数据建模与通信。

容器与 Kubernetes

  • 镜像（Image）：提供容器运行所需库、资源与配置的模板。
  • 容器（Container）：镜像的运行实例，构建一个隔离的应用环境。
  • Kubernetes：容器编排平台，负责管理与调度容器，实现从打包、部署到监控的自动化；支持容错重启、自动扩缩、服务发现/负载均衡、配置管理与健康探针。
  • Pod：K8s 的最小调度单位，可包含多个容器。
  • Node：实际运行 Pod 的主机（物理或虚拟），需运行容器运行时、kubelet 与 kube-proxy。
